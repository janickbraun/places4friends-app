## What this is

A native (iOS/Android) port of the existing **places4friends.com** Next.js web app:
a social map where friends recommend places (restaurants, bars, cafés, activities).
The UI is **German**. Almost every module is written for **web parity** — design
tokens, timestamp/initials/color formatting, map zoom levels, and Supabase query
shapes are deliberately mirrored from the web app so the two stay pixel- and
behaviour-compatible. When changing anything user-visible, preserve that parity;
comments throughout cite the web source being mirrored.

## Commands

```bash
npm start          # expo start (dev server; press i/a to open a build)
npm run ios        # expo run:ios     — build & run native iOS
npm run android    # expo run:android — build & run native Android
npm run web        # expo start --web
npm run lint       # expo lint (ESLint via eslint-config-expo)
```

There is **no test framework** configured — do not assume `npm test` exists.
There is no typecheck script either; run `npx tsc --noEmit` to check types (strict mode is on).

**Requires a development build, not Expo Go.** The app depends on native modules
(`react-native-maps`, Google Sign-In, `expo-notifications`) absent from Expo Go.
`src/lib/runtime.ts` exposes `isExpoGo` — use it to render graceful fallbacks
rather than crashing.

**`ios/` and `android/` are gitignored (Continuous Native Generation)** — they are
generated by `expo prebuild`, never committed. After pulling or adding anything
that changes native code (a new native dependency, an `app.json`/`app.config.ts`
plugin or Info.plist change), the installed binary is stale and the new native
module is simply missing at runtime (e.g. `Cannot find native module
'ExpoPushTokenManager'`). Regenerate before rebuilding:

```bash
npx expo prebuild --clean
```

## Environment

All client secrets are `EXPO_PUBLIC_*` (client-exposed by design; Supabase **RLS
is the security boundary**). Copy `.env.example` → `.env`. Required: Supabase URL
+ anon key, Google web/iOS OAuth client IDs, Mapbox public token, web `SITE_URL`
(for invite/legal links — normalized to an `https://` origin by
[src/lib/site.ts](src/lib/site.ts)), and the Google Maps Android SDK key.
`app.config.ts` injects the Android Maps key, the iOS permission strings and
`LSApplicationQueriesSchemes` on top of the static `app.json`.

**Real secrets live as Edge Function secrets**, never in `.env`: `RESEND_API_KEY`
/ `RESEND_FROM` (verification mail), `GEOAPIFY_API_KEY` (map snapshots),
`GOOGLE_PLACES_API_KEY` (place search), plus `SITE_URL` for the verify-email
redirect. `google-services.json` (Android FCM) is gitignored too — EAS cloud
builds get it from the `GOOGLE_SERVICES_JSON` file env var, which `app.config.ts`
prefers over the on-disk file.

## Architecture

**Routing** — `expo-router` file-based routing under `src/app/`. `typedRoutes`
and `reactCompiler` experiments are enabled. The provider stack in
[src/app/_layout.tsx](src/app/_layout.tsx) is
`GestureHandlerRootView → SafeAreaProvider → QueryClientProvider → AuthProvider`,
then the `(tabs)` group. The five tabs (`index` = the map/"Karte", `activities`,
`create`, `friends`, `profile`) render a custom [BottomNav](src/components/BottomNav.tsx)
instead of the default tab bar. Auth screens (`login`, `register`,
`reset-password`), `notifications`, `profile/[id]` and `profile/settings` live
outside the tab group. The root layout also routes notification taps (see
Notifications, below).

**Auth** — [AuthProvider](src/components/auth/AuthProvider.tsx) holds `{ user,
loading, emailVerified }` in context; the session persists in AsyncStorage and
auto-refreshes while the app is foregrounded (wired in
[src/lib/supabase.ts](src/lib/supabase.ts)). It also registers this device's push
token on sign-in and deletes it on `SIGNED_OUT`. Protected screens wrap their
content in [AuthGate](src/components/auth/AuthGate.tsx) (render-prop that yields
the authenticated `user`, or shows an `AuthPrompt` when signed out). Google and
Apple sign-in live in [SocialAuthButtons](src/components/auth/SocialAuthButtons.tsx)
— Google is native (`@react-native-google-signin`, **lazily imported** so the
screen still renders in Expo Go) feeding `signInWithIdToken`; email/password and
OAuth all go through the same Supabase client.

**Email verification** — `profiles.email_verified` is the source of truth (not
Supabase's own confirmation). `send-verification-email` (Resend, throttled 1/min,
token in `email_verifications`) mails a link to the public `verify-email`
function, which flips the flag and **302-redirects to `${SITE_URL}/verify-email
?status=…`** — Edge Functions can't serve HTML (the platform rewrites `text/html`
to `text/plain`), so the result page belongs on the website. The app picks the
flag up on the next `refreshVerification`; until then
[VerificationBanner](src/components/VerificationBanner.tsx) nags.

**Data layer** — one **shared** Supabase client (`@/lib/supabase`), unlike the web
app's per-request clients. Data access is organized one file per domain under
`src/lib/` (`activities`, `friends`, `map`, `places`, `createRecommendation`,
`profile`, `blocks`, `reports`, `settings`, `notifications`, `notificationFeed`);
UI components never call Supabase directly. **RLS scopes reads to the user + accepted friends**, so queries
deliberately omit manual network filtering (e.g. `fetchViewportPins` just filters
by bounds). Generated DB types live in
[src/types/database.types.ts](src/types/database.types.ts) — regenerate after
schema changes, don't hand-edit.

**State lives in three places**, and mixing them up causes bugs: **server state**
→ TanStack Query (all Supabase reads/caching); **auth session** → the AuthProvider
context; **client-only device preferences** → module-level `useSyncExternalStore`
stores backed by AsyncStorage — [mapLayer.ts](src/lib/mapLayer.ts) (`useMapLayer`)
and [onboarding.ts](src/lib/onboarding.ts). These stores are singletons so every
screen sees the same value live and it survives restarts; don't reach for context
or Query for them. Live badges/lists (`BottomNav`, the activities feed, friends,
the bell count) additionally subscribe to **Supabase realtime** `postgres_changes`
and refetch — note the `Math.random()` channel-name suffix in each, which guards
against Fast Refresh double-subscription; copy that idiom for new channels.

**Moderation (security-sensitive)** — blocking and reporting mirror the same
"RLS + `SECURITY DEFINER` RPC" model as friend invites. `blockUser`/`unblockUser`
([src/lib/blocks.ts](src/lib/blocks.ts)) call the `block_user`/`unblock_user`
RPCs, which atomically tear down any friendship/request in either direction and
record the block; RLS then hides the two users from each other (profiles, requests,
and each other's comments under mutual friends' posts). Reporting
([src/lib/reports.ts](src/lib/reports.ts), UI in
[ReportMenu](src/components/ReportMenu.tsx)) is an idempotent upsert on
`reports (activity_id, reporter_id)` — a repeat report is a silent no-op. Enforce
these boundaries server-side; don't add client shortcuts.

**Friend invites (security-sensitive)** — clients are forbidden by RLS from
inserting or self-accepting an `accepted` friendship. The only sanctioned path to
become friends via a link is the `SECURITY DEFINER` RPCs `validate_friend_invite_link`
and `accept_friend_invite`, called from [src/lib/friends.ts](src/lib/friends.ts).
Don't add a client-side shortcut that bypasses them.

Links are `${SITE_URL}/invite/<token>` and are the **only** path claimed as an
iOS universal link / Android app link (`associatedDomains` + `intentFilters` in
`app.json`); claiming `/profile/*` would also swallow the website's
`/profile/settings` and `/profile/friends` pages. `invite/[token].tsx` resolves
the token to its creator and forwards to that profile, where the existing accept
card redeems it. Opened while signed out, the token is parked in
[pendingInvite.ts](src/lib/pendingInvite.ts) and replayed by the auth screens, so
registering through an invite still lands on the inviter's profile. The
association files (`apple-app-site-association`, `assetlinks.json`) are hosted by
the **web** repo and must be deployed before a build ships `associatedDomains` —
iOS caches them at install time. Older `/profile/<id>?invite=<token>` links keep
working on both platforms.

**Friend suggestions** — the Freunde tab surfaces people to add from two
`SECURITY DEFINER` RPCs wrapped by [friendSuggestions.ts](src/lib/friendSuggestions.ts):
`get_friend_suggestions` (friends-of-friends, ranked by mutual count) and
`find_contacts_on_p4f` (address-book matching). Both bypass RLS to walk the
friendship graph, so **the exclusion set inside each function is the access
control** — they already drop yourself, anyone you have a friendship or request
with, blocked and banned users, and they return only columns the public
`profiles` SELECT policy exposes. Don't widen that shape. Contact matching hashes
e-mails on-device ([contacts.ts](src/lib/contacts.ts), lowercase hex SHA-256 to
match Postgres `encode(digest(...),'hex')`) and compares them against
`user_email_hashes`, a table with RLS on and *no* policies; raw contacts never
leave the device, the opt-in is explicit ([contactSync.ts](src/lib/contactSync.ts)),
and batches are capped at 1000 hashes server-side. Mobile-only for now — the web
app has no equivalent, a deliberate break from web parity.

**Maps** — `react-native-maps` with `supercluster` clustering.
[src/lib/map.ts](src/lib/map.ts) converts between RN-maps `Region`, lat/lng bounds,
and web tile zoom levels so search-result zoom matches the web's `getZoomLevelForType`.
`DEFAULT_REGION` is the centre of Germany. Tapping the active Karte tab fires a
`DeviceEventEmitter` event (`MAP_RESET_ZOOM_EVENT`) to clear search/selection and
zoom out, mirroring the web's `reset-map-zoom` window event. The base layer
(standard vs. "Satellit"/hybrid) is a persisted preference via `useMapLayer`
(see State, above). Opening directions goes through
[src/lib/navigation.ts](src/lib/navigation.ts), which offers Apple Karten and (on
iOS) Google Maps — the Google option requires the `comgooglemaps` entry in
`LSApplicationQueriesSchemes`, injected by `app.config.ts`.

**Recommendation map thumbnails** — on post creation, `generateMapSnapshot`
([src/lib/createRecommendation.ts](src/lib/createRecommendation.ts)) calls the
`generate-map-snapshot` Edge Function (Geoapify, server-side) to produce and cache
a static map image (`map_snapshot_url`). It's best-effort: failure returns `null`
and the post still saves, with [ActivityCard](src/components/ActivityCard.tsx)
falling back to a live map tile.

**Notifications** — two channels share one `send-push` Edge Function, fired
best-effort by `notifyPush` ([src/lib/notifications.ts](src/lib/notifications.ts))
after five social events (`new_place`, `comment`, `save`, `friend_request`,
`friend_accept`). The client only names the event + an id; the function
**validates it against real rows and resolves recipients with the service role**
so a client can't spam arbitrary pushes — keep it that way.
1. *Remote push* via Expo/APNs to recipients with `notifications_enabled` and a
   row in `push_tokens`.
2. *In-app feed (the bell)* — the same function inserts one `notifications` row
   per recipient regardless of the push toggle (RLS: read/update own, no client
   insert). Read via [notificationFeed.ts](src/lib/notificationFeed.ts), which
   deliberately imports **no** `expo-notifications`;
   [NotificationBell](src/components/NotificationBell.tsx) shows the live unread
   badge and `notifications.tsx` marks all read on open.

`expo-notifications` is loaded behind a lazy `require` guard in `notifications.ts`
so a missing native module degrades to a no-op instead of crashing at startup —
**all** `expo-notifications` access must funnel through that module. Tapping a
push routes via the root layout (friend events → Freunde, post events → feed),
for both in-session taps and cold starts. Push only works on a **real device**
(`Device.isDevice`), needs an APNs key in EAS (`eas credentials`), and TestFlight
uses the *production* APNs environment — `eas build --profile production` sets
that entitlement automatically.

**Place search** — `searchPlaces` calls the `places-search` Edge Function, which
prefers Google Places (when `GOOGLE_PLACES_API_KEY` is set as a function secret)
and otherwise falls back to Mapbox using the public token passed in the request
body. If the function itself is unreachable, the client falls back to a direct
Mapbox request, so search degrades gracefully.

**Account settings** ([src/lib/settings.ts](src/lib/settings.ts), UI at
`profile/settings`) mirrors the web SettingsView: profile/email/password updates,
the `notifications_enabled` toggle, a GDPR data export (RLS-scoped JSON written
with `expo-file-system` and handed to the OS share sheet — the native equivalent
of the web download), and account deletion through the `delete_own_user`
`SECURITY DEFINER` RPC.

**Supabase Edge Functions** (`supabase/functions/`, Deno runtime, excluded from the
app `tsconfig`): `places-search`, `generate-map-snapshot`, `send-push`,
`send-verification-email`, `verify-email`. Deploy with the Supabase CLI; they are
not part of the Metro bundle. All except `verify-email` require a JWT
(`verify_jwt = true`) and treat the caller as the acting user.

## Conventions

- **Styling**: NativeWind v4 (`className=` Tailwind on RN). Design tokens in
  [tailwind.config.js](tailwind.config.js) are ported from the web `globals.css`;
  brand green is `#226622`. Global CSS is imported at the top of the root layout.
- **DB naming quirk**: the `is_superlike` column is surfaced in the UI as
  "must-see" (`isMustSee`). `categories` is a Postgres `text[]`; `categories.ts`
  holds the canonical (German) category list, order-matched to the web.
- **Storage**: public buckets `avatars` and `activity-images`; `getAvatarUrl`
  resolves storage paths to public URLs (with optional cache-busting).
- **Path aliases**: `@/* → ./src/*`, `@/assets/* → ./assets/*`.
- **Imports**: VS Code is configured to auto-organize/sort imports and run
  `fixAll` on save; keep imports tidy to avoid lint churn.
- **Builds**: EAS profiles (`development`/`preview`/`production`) in `eas.json`;
  `appVersionSource: remote` with `autoIncrement` on production, so bump the
  marketing `version` in `app.json` (not the build number) for a release.
- **Native modules in Expo Go**: anything native that isn't needed at startup is
  loaded with a lazy `import()`/`require` behind a guard (Google Sign-In,
  `expo-notifications`, `expo-file-system`/`expo-sharing`). Follow that pattern —
  a top-level import of a native module crashes the whole app where it's absent.
